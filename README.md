# nnclinssoap

## Fork: Performance Improvements by Ivan Vogel


While building and testing the pipeline on the provided data, I came across two challenges that I tried to address:

- Long Docker build time
- Need to manually run two separate pipelines (scRNAseq, then Xenium spatial)

## Docker build time

I restructured the Docker build process:
- Minimized redundancy across libraries by introducing a common base image that installs the dependencies shared by all parts of the scRNAseq workflow (preprocessing, analysis, and spatialxenium containers all build on top of it now)
- CRAN packages are installed from binaries rather than compiled during the build
  - `rocker/r-ver`, used in the upstream implementation as base, already points at a binary CRAN repo (P3M)
  - Bioconductor packages still have to compile from source (P3M has no Linux binaries for Bioc), but I pointed them at P3M's Bioc mirror instead of bioconductor.org directly, since bioconductor.org's redirect was hanging and intermittently stalling builds
  - Dropped `BiocManager` and `dependencies=TRUE`, both unused
- **Result: base image build time went from ~360 min to ~33 min**
- Additional fixes:
  - Removed Quarto from the preprocessing/analysis images — installed but never used
  - Fixed the pbmcref download so the `.tar.gz` no longer gets left behind in the image layer (and bloats the image size)
  - Converted the Apptainer `.def` files to thin `docker-daemon` wrappers — the Docker image is now built first, and sif generation wraps around it
  - Fixed `build_all_images.sh` / `build_single.sh`, where a successful build was being reported as failed

## Pipeline integration

I integrated the pipeline so it can be run end-to-end in a single command, provided both scRNAseq and Xenium data are in place. I deployed both the original upstream implementation and the integrated version to AWS EC2 for testing (16 vCPUs, 128 GB RAM). Performance on the integrated version is comparable to running the scRNAseq and Xenium pipelines separately (currently ~33 min for the integrated version vs. ~36 min for the original upstream version) — **treat this as a preliminary comparison; more benchmarking is needed** (see TODOs).

**Example**
 
Original upstream version (assumes the scRNAseq whirl pipeline was already run, producing `seurat_object.rds`):
 
```bash
nextflow run main.nf \
    -profile test,docker \
    --outdir results \
    --reference_rds /data/nnclinssoap/scrnaseq/output/seurat_object.rds \
    --single_cell_label_col "predicted.celltype.l2"
```
 
Equivalent command for the integrated pipeline (`--scrnaseq_input` activates the scRNAseq branch):
 
```bash
nextflow run main.nf \
    -profile test,docker \
    --scrnaseq_input ../scrnaseq/configs/read_samples.txt \
    --outdir test_results_integrated \
    --single_cell_label_col "predicted.celltype.l2"
```


Summary of implementation changes:
- Built a scRNAseq workflow in Nextflow with four processes, chained together, producing an annotated Seurat file for label transfer:
  - `SCRNASEQ_PREPROCESS` — per sample, runs in parallel
  - `SCRNASEQ_INTEGRATE` — collects all samples, runs once (Harmony or RPCA)
  - `SCRNASEQ_ANNOTATE` — Azimuth or SingleR
  - `SCRNASEQ_DE` — edgeR pseudo-bulk DE
- Wrote new scripts in `bin/` rather than editing the existing ones in place, so the original whirl-based `scrnaseq/R/` scripts remain untouched for reference
  - Replaced YAML config loading with a small CLI arg parser
- Connected the processes together in `main.nf`:
  - Populating `--scrnaseq_input` activates the integrated pipeline. Providing `--reference_rds` retains the manual approach (i.e. assuming the rds was generated by the scRNAseq whirl pipeline)
  - Auto-resolves which metadata column to use for label transfer/DE depending on `annotation_method`
  - Made `LABEL_TRANSFER` channel-driven rather than parameter-dependent (assessed dynamically whether to execute label transfer or not)
- Moved BCO generation to the `nf-prov` plugin instead of whirl's biocompute writer — BCOs are now generated for the new scRNAseq processes as well, not just the original spatialxenium ones

While porting the R scripts over, I also found and fixed following:
- `cell_annotation.R` only defined `df`/`cell_count` inside the Azimuth branch but read them unconditionally afterward — would have crashed on SingleR. Fixed by computing summary stats inside each branch, and gave SingleR the same IQR print and label-count boxplot Azimuth gets
- SingleR's `FindMarkers`/`GetAssayData` calls needed a `JoinLayers()` call first (Seurat v5 splits layers after `merge()`) — likely never hit upstream since `annotation_method` defaults to Azimuth

## Overview

A comprehensive analysis framework for spatial transcriptomics data, with primary focus on 10x Genomics Xenium platform. The repository provides both Nextflow-based and R-based containerized workflows for processing and analyzing spatial single-cell data.

### Repository Structure

```
nnclinssoap/
├── docker/                    # Container definitions (Dockerfiles & Apptainer .def)
│   ├── preprocessing/         # QC, doublet detection, ambient RNA removal
│   ├── analysis/              # Cell annotation, differential expression
│   └── spatialxenium/         # Xenium-specific spatial analysis
│   └── qc-report/             # QC reporting for SpatialXenium
├── containers/                # Built Singularity images (.sif files)
├── SpatialXenium/             # Nextflow pipeline (nf-core style)
├── scrnaseq/                  # R-based pipeline using whirl
├── tests/                     # Testing scripts and validation
└── docs/                      # Documentation and guides
```

---

## Quick Start (Test case executed locally)

### 1. Download Test Data
```bash
./setup_test_data.sh
```

### 2. Build Containers
```bash
cd docker
./build_all_images.sh -v docker

# Builds 5 containers: base, preprocessing, analysis, spatialxenium, qc-report
# Total time: ~20 min (binary-package build via Posit P3M)
```

### 3. Run sc/snRNA-seq Pipeline
```bash
cd scrnaseq
./run_preprocessing_containerized.sh
./run_analysis_containerized.sh
```

### 4. Run SpatialXenium Pipeline
```bash
cd SpatialXenium
nextflow run main.nf -profile test,docker --outdir test_results
```

## Pipelines

### scrnaseq (R-based)

**Purpose:** Flexible R workflow for single-cell analysis  
**Framework:** whirl package  
**Workflows:**
- Preprocessing: QC, doublet detection, batch correction
- Analysis: Cell annotation, differential expression

**Documentation:** [`scrnaseq/README.md`](scrnaseq/README.md)

### SpatialXenium (Nextflow)

**Purpose:** Automated spatial transcriptomics analysis  
**Platform:** 10x Genomics Xenium  
**Features:**
- Quality control and filtering
- Spatial feature analysis
- Optional reference-based cell type annotation
- Comprehensive QC reporting

**Documentation:** [`SpatialXenium/README.md`](SpatialXenium/README.md)

## Container Images

All R-based containers use **R 4.5.2** with **Bioconductor 3.22**

| Container | Purpose | Key Packages | Size |
|-----------|---------|--------------|------|
| **preprocessing** | QC & batch correction | Seurat, SoupX, scDblFinder, harmony | ~8.09 GB |
| **analysis** | Annotation & DE | Azimuth, SingleR, edgeR, Signac | ~10.97 GB |
| **spatialxenium** | Spatial analysis | presto, Seurat, argparser, arrow, spatstat | ~7.95 GB |
| **qc-report** | QC PDF generation | Python 3.11, fpdf2 | ~0.3 GB |

## Troubleshooting

### Common Issues

**Container not found**
```bash
ls containers/*.sif
cd docker && bash build_all_images.sh
```

**Test data incomplete**
```bash
./setup_test_data.sh
find test_data/ -name "*.tar.gz"  # Should be empty
```

**Pipeline fails**
```bash
tail SpatialXenium/.nextflow.log
nextflow run main.nf -profile test,singularity -resume
```

**Full troubleshooting:** See [`TESTING_GUIDE.md`](docker/TESTING_GUIDE.md#common-issues-and-solutions)
